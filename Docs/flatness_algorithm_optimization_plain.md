# 平整度检测算法优化通俗说明

## 1. 先说明：不是从零重写算法

原来的平整度检测流程本身已经包含这些步骤：

1. 对 `env` 和 `mix` 做投影反射差分
2. 生成棋盘格增强图和 mask
3. 检测棋盘格角点
4. 左右角点匹配
5. 生成点云并计算平整度

所以这次不是“新增了这些步骤”，而是在原有步骤上做工程化增强。

可以理解成：

```text
原来：能按固定流程跑
现在：每一步多了兜底、质量判断、参数统一和兼容处理
```

下面按三个流程说明“原来怎么做”和“现在具体改了什么”。

## 2. 流程一：投影反射差分

### 2.1 图像对齐

原来的做法：

```text
主要使用 ECC + homography 做图像对齐
如果 ECC 失败，就直接返回原图
```

这意味着：只要 homography 这条路不稳定，就没有别的对齐尝试了。

这次改动：

```text
先尝试 homography
失败后尝试 affine
再失败尝试 translation
最后才退回 identity
```

也就是从“一次尝试”变成“多级兜底”。

区别不是“也做了对齐”，而是：

- 原来只试一种对齐模型
- 现在按复杂到简单依次尝试
- 复杂模型失败时，不会马上放弃
- 每次失败会写 warning 日志，方便知道是哪一级失败

这样做的意义是：如果图片纹理少、反光强、曝光差异大，homography 不一定能收敛，但 affine 或 translation 可能还能给一个可用结果。

### 2.2 ECC 输入预处理

原来的做法：

```text
把图转灰度，然后直接交给 ECC
```

这次改动：

```text
灰度化
高斯降噪
归一化到 0-1
限制迭代次数
捕获 OpenCV 异常
```

区别在于：ECC 对输入图像很敏感，如果亮度范围、噪声、高光差异比较大，直接跑容易失败。现在先把输入整理成更适合 ECC 的形式。

### 2.3 大图对齐速度

原来的做法：

```text
大图也直接在原尺寸上跑 ECC
```

这次改动：

```text
如果图片较大，先缩小图片估计变换矩阵
再把矩阵映射回原图尺寸
必要时再做少量精修
```

区别在于：

- 原来计算量跟原图大小强相关
- 现在大图先低分辨率估计，速度更快
- 仍然会把结果应用回原始尺寸，不改变输出尺寸

### 2.4 亮度补偿

原来的做法：

```text
用差异较小的区域做亮度拟合
每个颜色通道做线性补偿
```

这个思路本身没有问题。

这次改动主要在“哪些像素可以参与拟合”和“怎么拟合”：

原来参与拟合的像素筛选较简单：

```text
diff 小于阈值
排除过亮像素
```

现在筛选条件更细：

```text
排除明显变化区域
排除过亮区域
排除过暗区域
使用 median / MAD / percentile 估计更合理的差异阈值
mask 太小时退回可用亮度区域
```

也就是说，原来可能把高光、阴影、投影边缘混进去做亮度拟合；现在会尽量把这些点排除掉。

拟合方式也改了：

```text
原来：构造大矩阵，用 np.linalg.lstsq 拟合
现在：用一元线性回归闭式解，避免构造大矩阵
```

效果是：

- 结果含义不变，还是 `mix = beta * env + gamma`
- 内存开销更小
- 高分辨率图片时更轻量
- 有效样本太少时，用中位数偏移兜底

### 2.5 棋盘格 mask 处理

原来的做法：

```text
自适应阈值
闭运算
连通域过滤
膨胀
```

这次不是简单重复这些，而是补了几步：

```text
open 去小噪声
close 补断裂
填洞
按面积删除小连通域
只保留最大几个主要区域
再膨胀覆盖边缘
```

主要区别：

- 原来 mask 可能留下很多小块背景误检
- 现在会主动删除小区域
- 原来区域内部可能有洞
- 现在会填洞，让棋盘格区域更完整
- 原来所有超过面积的区域都可能保留
- 现在只保留主区域或少数最大区域

## 3. 流程二：棋盘格角点检测

### 3.1 检测策略

原来的做法：

```text
主要使用 findChessboardCornersSB
多尺度尝试
SB 失败后基本就失败
```

这次改动：

```text
先用 findChessboardCornersSB
失败后用 findChessboardCorners
并且对不同预处理图都试一遍
```

预处理版本包括：

```text
原图
CLAHE 对比度增强
blur 降噪
threshold 阈值化
sharpen 锐化
```

区别是：

- 原来主要是“同一个检测器换几个尺度”
- 现在是“多个检测器 + 多种预处理 + 多尺度”

这样低对比度、轻微模糊、边缘不清楚的图像，有更多机会被检测出来。

### 3.2 OpenCV 兼容

原来的问题：

不同 OpenCV 版本里：

- `CALIB_CB_PARTIAL_OK` 不一定存在
- `findChessboardCornersSB` 返回格式可能不同

这次处理：

```text
使用 getattr 检查 flag 是否存在
统一封装 OpenCV 返回结果
不假设返回格式固定
```

这不是算法层面的新东西，但能避免换环境后代码突然报错。

### 3.3 角点质量判断

原来的做法：

```text
allow_partial=True 时，只要数量超过 50% 就接受
```

问题是：数量够，不代表角点一定可靠。

这次新增检查：

```text
角点数量
覆盖区域
重复点
最近邻距离
完整棋盘格的行列间距
完整棋盘格的排序单调性
```

但是这里后来做了一个重要修正：

一开始我把覆盖率当成硬条件，导致你的数据出现：

```text
coverage=0.0087
```

就被拒绝。

这其实不合理，因为棋盘格在整张图里占比小，不代表角点错了。

现在改成：

```text
如果检测到了完整棋盘格，不再因为覆盖率略小直接失败
如果只是部分角点，才用覆盖率防止明显误检
```

也就是说，完整角点优先相信几何结构，而不是只看占图面积。

### 3.4 亚像素精化

原来的做法：

```text
直接调用 cornerSubPix
```

这次改动：

```text
先统一图像类型
先统一角点格式
如果 cornerSubPix 报 OpenCV 异常，就保留原角点继续走
```

区别是：精化失败不会让整个检测流程崩掉。

## 4. 流程三：左右角点匹配

### 4.1 网格排序方式

原来的做法：

```text
按 y 坐标排序分行
每一行按 x 坐标排序
左右图按相同行列位置匹配
```

这种方式在理想正视图里比较直观，但遇到透视变形、旋转、缺失角点时，单纯按 y 分行容易错。

这次改动：

```text
先根据所有角点估计棋盘格整体方向
再把角点投影到行方向和列方向
最后分配到固定网格
```

也就是从：

```text
只看 y/x 坐标
```

变成：

```text
先看棋盘格整体朝向，再判断行列
```

这可以减少因为棋盘格倾斜、透视造成的排序错位。

### 4.2 缺失角点处理

原来的做法已经有 NaN 占位思想。

这次保留这个兼容方式，但调整了网格分配逻辑：

```text
每个网格位置只保留最接近该网格中心的角点
缺失位置继续用 NaN
左右图只匹配双方都有效的位置
```

这样不会因为某一侧少几个点，就把后面所有点整体错位。

### 4.3 几何质量检查

这次新增了匹配质量检查：

```text
视差均值
视差标准差
视差离群点
y 坐标差异常点
匹配比例
异常点数量
```

并输出到：

```text
match_quality.json
```

但这里也做了一个重要修正。

一开始我把几何异常当成硬失败，所以你遇到：

```text
异常匹配比例过高 100%
```

就直接中断。

这对你原来的数据不兼容，因为你的图可能不是严格校正后的双目图，或者视差方向/y 差不符合我假设的标准双目几何。

现在默认改成兼容模式：

```text
异常比例高：记录 warning
继续保留匹配点
继续输出 corners_left.json 和 corners_right.json
在 match_quality.json 里记录异常数量
```

如果以后真的要严格拦截，可以在配置里打开：

```text
strict_geometry=True
```

所以区别是：

- 原来：没有详细质量指标
- 第一次优化：质量异常直接失败，太严格
- 现在：质量异常只提示，不破坏旧流程

## 5. 配置文件具体解决什么问题

原来很多参数直接写在代码里，比如：

```text
棋盘格尺寸
ECC 迭代次数
差分阈值
mask 面积阈值
角点比例阈值
匹配异常阈值
```

这次新增：

```text
Algorithm/algorithms/flatness_detection/config.py
```

把这些参数集中起来。

这样做的区别是：

- 原来调参数要在多个文件里找
- 现在大部分参数集中在一个配置文件
- 默认值保持旧流程能跑
- 后续如果服务层要传自定义配置，也有入口

## 6. 输出文件有什么变化

原来的输出保持：

```text
left_detect.png
right_detect.png
left_mask.png
right_mask.png
corners_left.json
corners_right.json
```

新增：

```text
match_quality.json
```

这个文件不是给旧流程必须读取的，而是给调试和判断质量用。

例如可以看：

```text
matched_count：实际匹配点数量
expected_count：理论角点数量
match_ratio：匹配比例
disparity_mean：平均视差
disparity_std：视差波动
outlier_count：异常匹配点数量
```

## 7. 这次修正了两个误杀问题

### 7.1 角点覆盖率误杀

之前失败：

```text
角点覆盖区域过小：coverage=0.0087
```

现在改为：

```text
完整角点不因为覆盖率小直接失败
只对部分角点使用覆盖率强约束
```

### 7.2 匹配几何异常误杀

之前失败：

```text
异常匹配比例过高 100%
```

现在改为：

```text
默认兼容模式下只 warning
不直接中断流程
继续输出角点结果
```

这两个改动的目的都是：不要让新增质量检查破坏旧数据可运行性。

## 8. 测试覆盖了什么

新增测试文件：

```text
Algorithm/tests/test_flatness_front_flows.py
```

覆盖场景包括：

- ECC 失败 fallback
- 输入图片尺寸不同
- 亮度变化明显
- mask 小区域过滤
- 正常棋盘格角点检测
- 低对比度棋盘格角点检测
- 完整角点但覆盖率小，不误杀
- `cornerSubPix` 失败 fallback
- 完整左右角点匹配
- 缺失部分角点匹配
- 少量异常匹配剔除
- 大量异常匹配默认兼容通过
- 严格模式下大量异常匹配可以失败
- `detect.main(data_dir, result_dir)` 保持旧调用兼容

运行命令：

```powershell
python -m unittest discover -s Algorithm\tests -v
```

目前结果：

```text
15 tests OK
```

## 9. 总结一句话

原来的算法已经有完整流程。

这次真正做的不是“加了差分、加了角点检测、加了匹配”，而是：

```text
把原来单一路径的流程，改成多兜底、多校验、可配置、可观察，并且默认不破坏旧数据运行。
```

后续如果要进一步提升平整度数值准确性，还需要真实相机标定参数、双目标定和更多真实数据调参。这部分不是本次前 3 流程优化的重点。
